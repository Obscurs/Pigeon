# ADR 0012 — In-engine image matting for figure compositing (onnxruntime CUDA + isnet)

**Status:** Accepted
**Date:** 2026-06-26

## Context

ADR 0011 composites the generated figure over the restyled background through a **soft alpha
mask built from the OpenPose skeleton's silhouette** (`m_CompositeWithSkeletonMask`): a union of
thick limb sticks, a torso block, and a head disc, box-blurred into a feathered edge. That mask is
**geometric** — derived from the skeleton's bounds, not from the rendered character — so it is
necessarily larger than and differently-shaped from the actual figure. Wherever the mask is opaque
but the underlying pixel is the figure's own plain generated background (the gaps between limbs, the
torso block overhanging the body, the feathered margin), that flat background is blended onto the
scene. The visible result is a "gray cloud" halo hugging the character, and it is **not tunable
away**: `m_CompositeMaskScale` only scales the stroke width, so tightening it clips the body and
loosening it re-floods the halo. The alpha can never match the true silhouette because it does not
look at the pixels.

The fix is a **pixel-accurate cutout**: an Alpha Matte produced by a real image-matting model that
follows the character's actual outline, used as the composite alpha in place of the skeleton
silhouette. This requires running a matting model in-process at generation time (the figure only
exists at runtime), which collides with a hard fact of the engine: **the entire in-engine ML stack
is GGML** — stable-diffusion.cpp (diffusion, CUDA) and llama.cpp (text-gen, CUDA) each in an
isolated `ggml` build scope (ADR 0008/0009/0010). Off-the-shelf matting/segmentation models
(u2net, isnet, BiRefNet, SAM) are distributed as **ONNX**, with no GGML-native equivalent. So a
real matte means introducing a **second inference-runtime family** (ONNX) alongside GGML.

## Decision

Add a **Matting Backend** — a platform-abstracted ONNX inference runtime — and composite the figure
through its Alpha Matte **inline in the diffusion worker**, replacing the skeleton-mask alpha branch.

- **Runtime: onnxruntime with the CUDA execution provider, falling back to CPU.** It shares the GPU
  with the resident SDXL checkpoint (the matte is small, ~tens of MB working set) and reuses the CUDA
  toolchain diffusion/text-gen already require, rather than introducing a third GPU API (DirectML).
  onnxruntime ships **prebuilt GPU binaries**, so there is no monster source build — but they are
  **DLLs**, making this the one engine dependency *not* linked statically. **Crucial caveat:** the CUDA
  EP additionally needs **cuDNN**, which the GGML CUDA stack (cuBLAS/cudart only) does *not* ship — and the
  onnxruntime GPU zip is built for a SPECIFIC CUDA major, so a machine that runs diffusion on the GPU may
  still fail to load `onnxruntime_providers_cuda.dll` (`LoadLibrary` error 126 = missing dependent DLL) on
  both counts. Resolved on two fronts: (1) the build pins the **CUDA-12** onnxruntime build (1.20.1) to
  match the GGML toolkit's CUDA 12, and **bundles the cuDNN 9 redist DLLs** next to the executable (fetched
  from NVIDIA's public redist, ~800 MB) so the CUDA EP loads out of the box; (2) the backend still **catches
  CUDA-EP init failure and recreates the session on the CPU EP** (isnet is ~1–2s on CPU) as a safety net, so
  a toolkit mismatch degrades gracefully instead of silently dropping to the skeleton fallback. Bump the
  onnxruntime and cuDNN versions together on the same CUDA major.
- **Model: isnet-general-use.** A prompt-free saliency-matting model (the modern rembg default),
  declared in a Resource Manifest's `mattingModels` array and loaded from the `ImageGeneration`
  folder. Like a Checkpoint, the resource map records only its resolved **path**
  (`m_MattingModelMap`); the backend loads the weights. One model, startup-resident.
- **Inline in the diffusion worker.** The figure is still generated by the proven txt2img + OpenPose
  ControlNet + character-LoRA path on a plain background; the existing worker, after `Generate()`,
  calls the resident Matting Backend to obtain the Alpha Matte of the generated figure and composites
  it over the restyled background with the existing `MaskComposite` blend (the matte replaces the
  skeleton mask as that blend's alpha source). The composite stays a CPU blend off the main thread;
  only the alpha's *origin* changes.
- **Graceful fallback to the skeleton silhouette.** When the Matting Backend is unavailable — the
  Testing/UT build (`PG_ENABLE_MATTING=OFF`, the inert no-op, mirroring how diffusion/text-gen are
  disabled there), the opt-in off, or the model file missing — the worker falls back to ADR 0011's
  skeleton-silhouette mask. The matte is an *upgrade to the alpha source*, not a new hard dependency
  for the pipeline to function, so UT remains deterministic with no ONNX/CUDA present.

Mirroring DiffusionBackend / TextGenBackend: `pg::MattingBackend` (`LoadModel`/`IsLoaded`/`Matte`),
`Create()` selecting **OnnxMattingBackend** (real, behind `PG_ENABLE_MATTING` → `PG_MATTING_ENABLED`)
or **TestingMattingBackend** (deterministic mock), held in a `MattingBackendSingletonComponent` that
`DiffusionSystem` creates and loads once at startup alongside the diffusion backend. The app opts a
generation into the matte composite with a new `m_CompositeWithMatte` flag on the Generate Image
Request; SandboxApp's `ImageGenDemoSystem` sets it for step 3 in place of `m_CompositeWithSkeletonMask`.

## Trade-offs

- **onnxruntime + CUDA (chosen) vs onnxruntime + DirectML (rejected):** DirectML is Windows/DX11-native
  and needs no CUDA, but it would add a *third* GPU runtime to a stack already standardized on CUDA for
  both GGML backends. CUDA EP keeps one GPU API and one driver/toolkit surface. Cost: onnxruntime's CUDA
  EP is **version-coupled** to specific CUDA/cuDNN releases, which must agree with the toolkit sd.cpp and
  llama.cpp build against (a pinned, verified onnxruntime version, like the pinned sd.cpp/llama commits).
- **A real matte (chosen) vs masked chroma-key (rejected):** the figure is generated on a plain
  background, so a skeleton-bounded colour key would also have removed the halo with **zero new
  dependencies**. It was rejected because keying is brittle on an anime figure whose light clothing/skin
  collides with a light key colour, and the point of "integrate the character into the real background"
  is edge quality (hair, fingers) a colour key cannot deliver. The matte is the heavier but
  qualitatively-correct path; the user chose it explicitly over the cheap fix.
- **onnxruntime/ONNX (chosen) vs a ggml-native segmenter — sam.cpp (rejected):** a ggml port would keep
  a single runtime family, but sam.cpp needs point/box prompts (not a direct matte), and a hand-written
  u2net/isnet ggml graph is far more code than wiring a prebuilt ONNX session. The architectural cost of
  a second runtime family was judged smaller than re-implementing the model.
- **Inline in the diffusion worker (chosen) vs a separate Matting System + step (rejected):** a distinct
  step would match ADR 0011's "separately-inspectable stages" and surface the raw matte on screen, but it
  adds a system, a request type, and a 4th app step for a cheap, deterministic post-step of an existing
  job. Inline keeps the change to "swap the alpha source," at the cost of coupling the Matting Backend
  into `DiffusionSystem` and not exposing the matte as its own texture.

## Consequences

- New engine surface: `pg::MattingBackend` (Diffusion module) + `OnnxMattingBackend`/`TestingMattingBackend`
  (`Platform/Onnx`), `MattingBackendSingletonComponent`, `ResourceMapSingletonComponent.m_MattingModelMap`,
  a `mattingModels` manifest category + `ResourceManagerSystem` loader, and a `m_CompositeWithMatte` field
  on `GenerateImageRequestOneFrameComponent`. `DiffusionSystem` gains the matting backend in its
  write/add sets and the matte call in its worker.
- **First non-static dependency.** onnxruntime GPU ships as DLLs (`onnxruntime.dll`,
  `onnxruntime_providers_cuda.dll`, `onnxruntime_providers_shared.dll`), copied next to the executable by
  a post-build step — an explicit exception to the "link everything statically" rule, gated behind
  `PG_ENABLE_MATTING`.
- **First ONNX runtime in a GGML stack.** onnxruntime vendors no `ggml`, so it does not collide with the
  isolated sd.cpp/llama ggml scopes; it does add a CUDA/cuDNN-version constraint shared with them.
- The Testing/UT build sets `PG_ENABLE_MATTING=OFF` (alongside diffusion/text-gen off), so UT compiles and
  runs with no ONNX/CUDA; the skeleton-silhouette fallback keeps the composite path covered there.
- SandboxApp's `ImageGenDemoSystem` step 3 switches to the matte composite; the "Composite mask tightness"
  knob remains meaningful only on the fallback path.
